## 3. Layout Strategy

```
BaseLayout.astro
│  <html>, <head>, fonts, global meta, OG tags, canonical
│  <slot /> for everything below
│
├── DefaultLayout.astro
│   │  <Header /> + <slot /> + <Footer />
│   │  Used by: Homepage, About, Careers, Contact, Request Proposal,
│   │           Knowledge Hub index, Case Studies index, Client Portal
│   │
│   ├── ArticleLayout.astro
│   │   Article meta bar (cluster / read time / date) + 2-col (body | sidebar)
│   │   Sidebar: TOC (vanilla JS scrollspy) + RelatedLinks
│   │   Used by: all 30 articles via [...slug].astro
│   │
│   ├── AuthorityPageLayout.astro
│   │   AuthorityHero + 2-col body (main | sticky sidebar)
│   │   Sidebar: Related articles + related authority pages + CTA
│   │   Used by: all 10 authority .astro files
│   │
│   ├── CaseStudyLayout.astro
│   │   OutcomeStrip header (full-width dark band) + body content
│   │   Used by: all 4 case studies via [...slug].astro
│   │
│   └── DDHubLayout.astro
│       DDNavBar (sticky sub-nav) + page content
│       Used by: all 6 DD Hub pages
```

**Key decisions:**
- Authority pages are `.astro` files (not MDX) — their layouts are too complex and multi-columnar for MDX to express cleanly
- Articles and case studies are MDX — pure content with predictable structure
- `BaseLayout` handles all `<head>` concerns; child layouts never touch `<head>`

## 4. Content Collections Schema

### Articles (`src/content/articles/*.mdx`)

```typescript
// src/content/config.ts

import { defineCollection, z } from 'astro:content'

const clusterEnum = z.enum([
  'government',
  'industrial', 
  'education',
  'carbon',
  'procurement',
  'technology'
])

const articles = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    cluster: clusterEnum,
    articleCode: z.string(),                          // 'G1', 'I3', etc. — encodes cluster + number
    authorityPage: z.string(),                        // 'A1' slug or full slug
    authorityPageUrl: z.string(),
    primaryKeyword: z.string(),
    metaDescription: z.string().max(160),
    publishDate: z.date(),
    lastUpdated: z.date().optional(),
    readTime: z.number().int(),                       // minutes
    relatedArticles: z.array(z.string()).default([]), // article slugs
    relatedAuthority: z.array(z.string()).default([]),
    featured: z.boolean().default(false),
  }),
})

const casestudies = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    code: z.string(),           // 'CS1'–'CS4'
    sector: z.string(),         // 'Government', 'Industrial', etc.
    sectorCluster: clusterEnum,
    location: z.string(),       // 'QLD', 'QLD + NSW + VIC + WA', etc.
    contractTerm: z.string(),   // '3-year contract', etc.
    outcomes: z.array(z.object({
      label: z.string(),
      value: z.string(),
      sub: z.string().optional(),
    })).min(3).max(6),
    services: z.array(z.string()),
    quote: z.object({
      text: z.string(),
      attribution: z.string(),
    }).optional(),
    metaDescription: z.string().max(160),
    relatedAuthority: z.array(z.string()).default([]),
    relatedArticles: z.array(z.string()).default([]),
    featured: z.boolean().default(false),
  }),
})

export const collections = { articles, casestudies }
```

## 5. Brand Tokens (Tailwind Config)

```javascript
// tailwind.config.mjs
export default {
  theme: {
    extend: {
      colors: {
        navy:    { DEFAULT: '#0a0f1a', 800: '#111827', 700: '#1e293b' },
        // Teal: PRIMARY accent — CTAs, headings, stats, grid overlay (wireframes A–F)
        teal:    { DEFAULT: '#00d4aa', bright: '#00f0c4', dark: '#00a882' },
        // Gold: TERTIARY highlight only — used sparingly for key differentiator callouts
        gold:    { DEFAULT: '#f59e0b' },
        blue:    { DEFAULT: '#3b82f6' },
        text:    { primary: '#374151', muted: '#94a3b8', light: '#e2e8f0' },
      },
      fontFamily: {
        display: ['Bebas Neue', 'sans-serif'],
        body:    ['DM Sans', 'sans-serif'],
      },
    },
  },
}
```

**Usage rules:**
- Teal (`#00d4aa`) — PRIMARY: all CTAs, heading accents, grid overlays, stats numbers, proof boxes, KPI callouts, Knowledge Hub chrome. This is the dominant accent colour throughout.
- Gold (`#f59e0b`) — TERTIARY only: used sparingly for specific "key differentiator" callouts where the wireframes show it. Not used for CTAs or headings.
- Navy `#0a0f1a` — hero backgrounds, dark section bands, header/footer

## 6. Build Phases (12 core + Phase 8b: Locations)

### Phase 1 — Scaffold (AUTONOMOUS)
**Goal:** Working Astro project on localhost, committed to GitHub

- `npm create astro@latest . -- --template minimal --typescript strict --no-install`

- `src/styles/fonts.css`: @font-face for Bebas Neue + DM Sans (from `/public/fonts/`)

- `BaseLayout.astro`: html shell, head, fonts link, slot
- `.gitignore`: node_modules, dist, .env

Deliverable: Astro scaffold running on localhost, committed and pushed to GitHub

## 11. AI Authority Requirements

**Strategic purpose:** Every page on this site must be citable by AI search engines (ChatGPT, Perplexity, Claude, Google AI Overviews, Gemini, Microsoft Copilot). The 30 articles, 10 authority pages, and case studies exist so that when procurement officers ask AI assistants about commercial cleaning in Australia, CPC is the answer — with citations. These requirements are enforced throughout the build, not deferred to Phase 10.

---

### 11.1 Content Structure (all articles, authority pages, case studies)

- **First 100 words answer a clearly-stated question or claim.** No scene-setting. AI engines read the first paragraph to determine if the page is a valid source.
- **TL;DR / Quick Answer block** at the top of every article and authority page — 50–80 words, plain language, directly answering the page's primary question. Marked with `<SpeakableSpecification>` schema to signal to voice and AI assistants.
- **Headings written as natural-language questions** where possible (e.g. "What Standards Apply in Government Cleaning Contracts?" not "Standards Overview"). Matches AI query patterns directly.
- **Concrete numbers, named clients (where permitted), specific standards, dates.** AI engines preferentially cite content with verifiable specificity over generic prose.
- **Proprietary CPC data** (97–99% audit pass rates, 0 LTI records, 12+ government sites, +280 renewable energy credits, +2350 trees planted) presented with date and methodology note. Original data is preferentially cited over rehashed industry stats.
- **WHO / WHERE / WHAT in first paragraph** of every page: state "Commercial Property Cleaning (CPC)" by full name, specific Australian regions served, and specific service. Entity clarity is how AI engines build knowledge graph associations.
- **External authority links** where CPC content references standards (AS/NZS, ISO, Climate Active, Modern Slavery Act 2018) — link to `.gov.au` and standards bodies. AI engines cross-reference these signals for topical authority.
- **`dateModified` frontmatter** on every article and case study, rendered as a visible "Last updated" line. Content freshness is weighted by AI engines.
